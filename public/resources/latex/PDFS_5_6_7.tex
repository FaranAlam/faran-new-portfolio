\section{Git Fundamentals}
% ============================================

\begin{infobox}{Why Git is Non-Negotiable}
Git is the most important tool a developer must master. It tracks every change, enables team collaboration, allows you to undo mistakes, and is required by every company. No Git = no job. Simple.
\end{infobox}

\subsection{Git Architecture}

\begin{center}
\begin{tabularx}{\textwidth}{|>{\bfseries\color{FaranRed}}l|X|}
\hline
\rowcolor{FaranDark}
\color{white}\textbf{Area} & \color{white}\textbf{Description} \\
\hline
Working Directory & Your actual files where you write code \\
\hline
\rowcolor{red!5}
Staging Area & Files added with \texttt{git add} -- ready to commit \\
\hline
Local Repository & Committed snapshots on your machine \\
\hline
\rowcolor{red!5}
Remote Repository & GitHub/GitLab -- shared cloud copy \\
\hline
\end{tabularx}
\end{center}

\subsection{Essential Git Commands}

\begin{codebox}{Daily Git Workflow}
\begin{lstlisting}
# Setup (one time)
git config --global user.name "Faran Alam"
git config --global user.email "faran.bsce40@iiu.edu.pk"

# Start project
git init                          # new repo
git clone <url>                   # clone existing

# Daily workflow
git status                        # what changed?
git add .                         # stage all changes
git add filename.js               # stage one file
git commit -m "feat: add login"   # save snapshot
git push origin main              # push to GitHub
git pull origin main              # get latest changes
\end{lstlisting}
\end{codebox}

\begin{codebox}{Branching \& Merging}
\begin{lstlisting}
# Branches
git branch                        # list branches
git branch feature/login          # create branch
git checkout feature/login        # switch branch
git checkout -b feature/dashboard # create + switch

# Merging
git checkout main
git merge feature/login           # merge into main
git branch -d feature/login       # delete merged branch

# Pull Requests (GitHub flow)
# 1. Create feature branch
# 2. Push to GitHub
# 3. Open Pull Request
# 4. Review + merge
\end{lstlisting}
\end{codebox}

\begin{codebox}{Undoing Mistakes}
\begin{lstlisting}
# Undo last commit (keep changes)
git reset --soft HEAD~1

# Undo last commit (discard changes)  
git reset --hard HEAD~1

# Undo changes to a file
git checkout -- filename.js

# See commit history
git log --oneline

# Go back to specific commit
git checkout abc1234

# Revert a commit (safe for public repos)
git revert abc1234
\end{lstlisting}
\end{codebox}

\begin{warningbox}
\textbf{NEVER} force push to main/master on a shared repository. \texttt{git push --force} overwrites history and can permanently destroy teammates' work. Only use it on your personal feature branches when absolutely necessary.
\end{warningbox}

\section{CI/CD Pipelines}
% ============================================

\begin{infobox}{What is CI/CD?}
\textbf{CI (Continuous Integration)} = Automatically run tests every time you push code.\\[0.2cm]
\textbf{CD (Continuous Deployment)} = Automatically deploy to production if tests pass.\\[0.2cm]
Result: No more manual deployments. Push code → Tests run → Auto deploy. Zero human error.
\end{infobox}

\subsection{GitHub Actions (Free CI/CD)}

\begin{codebox}{.github/workflows/main.yml}
\begin{lstlisting}
name: Deploy to Production

on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest

    steps:
      # 1. Get the code
      - uses: actions/checkout@v3

      # 2. Setup Node.js
      - uses: actions/setup-node@v3
        with:
          node-version: '18'
          cache: 'npm'

      # 3. Install dependencies
      - run: npm ci

      # 4. Run tests
      - run: npm test

      # 5. Build project
      - run: npm run build

      # 6. Deploy (Vercel / Railway auto-detects pushes)
\end{lstlisting}
\end{codebox}

\subsection{Production Deployment Checklist}

\begin{itemize}[leftmargin=1.5em, itemsep=6pt]
    \item[\ding{52}] Environment variables set in platform dashboard (never in code)
    \item[\ding{52}] Database connection string is production DB, not local
    \item[\ding{52}] \texttt{NODE\_ENV=production} is set
    \item[\ding{52}] Error logging enabled (Sentry, LogRocket)
    \item[\ding{52}] HTTPS enforced (no HTTP traffic)
    \item[\ding{52}] CORS configured for production domain only
    \item[\ding{52}] Rate limiting enabled
    \item[\ding{52}] API keys rotated from development values
    \item[\ding{52}] Build passes locally: \texttt{npm run build}
    \item[\ding{52}] Custom domain connected with SSL certificate
\end{itemize}

\begin{tipbox}
For Next.js projects, Vercel is the simplest choice -- it's built by the same team. Connect your GitHub repo, add environment variables, done. Every push to main auto-deploys in under 60 seconds.
\end{tipbox}

\section{SQL vs NoSQL: The Real Difference}

\begin{center}
\begin{tabularx}{\textwidth}{|X|X|}
\hline
\rowcolor{FaranDark}
\color{white}\large\textbf{SQL (Relational)} & \color{white}\large\textbf{NoSQL (Document)} \\
\hline
Tables with rows \& columns & Collections with documents (JSON) \\
\hline
\rowcolor{teal!5}
Strict schema (defined upfront) & Flexible schema (schema-less) \\
\hline
JOINS between tables & Embedded documents or references \\
\hline
\rowcolor{teal!5}
ACID transactions guaranteed & Eventual consistency (configurable) \\
\hline
Great for: finance, e-commerce & Great for: content, real-time, flexible \\
\hline
\rowcolor{teal!5}
PostgreSQL, MySQL, SQLite & MongoDB, Firebase, DynamoDB \\
\hline
\end{tabularx}
\end{center}

\begin{infobox}{When to Use Which?}
\textbf{Use SQL when:} Data is highly relational (users-orders-products), you need complex reporting, financial accuracy is critical.\\[0.2cm]
\textbf{Use MongoDB when:} Building fast MVPs, data structure varies, you want to store documents/JSON naturally, or working with Next.js/Node.js.
\end{infobox}

\subsection{MongoDB Mongoose Example}

\begin{codebox}{User Schema with Mongoose}
\begin{lstlisting}
const mongoose = require('mongoose');
const bcrypt = require('bcryptjs');

const userSchema = new mongoose.Schema({
  name: {
    type: String,
    required: [true, 'Name is required'],
    trim: true,
    maxlength: [50, 'Name too long']
  },
  email: {
    type: String,
    required: true,
    unique: true,
    lowercase: true,
    match: [/\S+@\S+\.\S+/, 'Invalid email']
  },
  password: {
    type: String,
    required: true,
    minlength: 8,
    select: false  // never return in queries
  },
  role: {
    type: String,
    enum: ['user', 'admin'],
    default: 'user'
  },
  createdAt: { type: Date, default: Date.now }
});

// Hash password before saving
userSchema.pre('save', async function(next) {
  if (!this.isModified('password')) return next();
  this.password = await bcrypt.hash(this.password, 12);
});

module.exports = mongoose.model('User', userSchema);
\end{lstlisting}
\end{codebox}

\section{Database Performance Tips}

\subsection{MongoDB Indexing}
\begin{itemize}[leftmargin=1.5em, itemsep=6pt]
    \item[\ding{52}] Index fields you query frequently: \texttt{userSchema.index(\{ email: 1 \})}
    \item[\ding{52}] Compound index for multi-field queries
    \item[\ding{52}] Use \texttt{.explain('executionStats')} to analyze query performance
    \item[\ding{52}] Avoid \texttt{\$or} on non-indexed fields -- very slow
    \item[\ding{52}] Use \texttt{.select('-password')} to exclude heavy fields
    \item[\ding{52}] Paginate large results: \texttt{.skip(page * limit).limit(limit)}
\end{itemize}

\begin{tipbox}
MongoDB Atlas (free tier: 512MB) is perfect for learning and small production apps. It includes a visual query profiler, backups, and global clusters. Use it instead of local MongoDB for all deployed projects.
\end{tipbox}

\section{Technical Interview Preparation}

\subsection{JavaScript \& React - Must Know Questions}

\begin{qbox}{What is the difference between var, let, and const?}
\begin{itemize}[leftmargin=1.2em, noitemsep]
    \item \textbf{var}: function-scoped, hoisted, old -- \textit{avoid using}
    \item \textbf{let}: block-scoped, reassignable, modern standard
    \item \textbf{const}: block-scoped, cannot be reassigned (but objects/arrays can be mutated)
\end{itemize}
\end{qbox}

\begin{qbox}{What is closure in JavaScript?}
A closure is when an inner function has access to the variables from its outer function's scope, even after the outer function has returned.\\[0.2cm]
\texttt{function counter() \{ let count = 0; return () => ++count; \}}\\
\texttt{const inc = counter(); inc(); // 1, inc(); // 2}
\end{qbox}

\begin{qbox}{Explain Event Loop in JavaScript}
JavaScript is single-threaded. The Event Loop allows async operations:\\
1. \textbf{Call Stack} -- executes synchronous code\\
2. \textbf{Web APIs} -- browser handles async (setTimeout, fetch)\\
3. \textbf{Callback Queue} -- completed async callbacks wait here\\
4. \textbf{Microtask Queue} -- Promises (higher priority than callbacks)\\
5. Event Loop moves tasks from queues to stack when stack is empty
\end{qbox}

\begin{qbox}{What is React reconciliation?}
When state/props change, React creates a new Virtual DOM tree and compares it with the previous one (diffing algorithm). Only the minimal number of real DOM updates are applied. This is why React is fast -- real DOM operations are expensive.
\end{qbox}

\begin{qbox}{Explain useEffect cleanup}
Cleanup prevents memory leaks. Return a function from useEffect to run when component unmounts or before next effect:\\[0.2cm]
\texttt{useEffect(() => \{ const sub = subscribe(); return () => sub.unsubscribe(); \}, [])}
\end{qbox}
